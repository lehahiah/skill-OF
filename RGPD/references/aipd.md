# AIPD - repères pratiques pour les organismes de formation

## Objet de cette référence

Cette référence a pour but d'aider une IA spécialisée RGPD à :
- repérer les situations dans lesquelles une analyse d'impact relative à la protection des données (AIPD) doit être envisagée ;
- raisonner avec prudence sur la notion de risque élevé ;
- éviter les affirmations automatiques ou excessives ;
- produire une analyse structurée, nuancée et juridiquement prudente.

Cette référence ne remplace pas :
- l'examen du traitement concret ;
- l'analyse du contexte réel ;
- la consultation du DPO lorsqu'il existe ;
- l'examen des textes et positions officielles applicables à la date de la demande.

---

## 1. Règle générale

L'AIPD est prévue par l'article 35 du RGPD. Elle est requise lorsqu'un traitement est susceptible d'engendrer un risque élevé pour les droits et libertés des personnes physiques.

L'analyse ne doit jamais reposer sur une intuition générale ou sur une catégorie abstraite de traitement. Elle doit reposer sur :
- la description concrète du traitement ;
- les catégories de données traitées ;
- les personnes concernées ;
- l'échelle du traitement ;
- le caractère systématique ou non du dispositif ;
- les effets potentiels sur les personnes ;
- les garanties déjà mises en place.

---

## 2. Position attendue de la skill

La skill ne doit pas affirmer automatiquement qu'une AIPD est obligatoire du seul fait que le traitement concerne :
- des salariés ;
- des apprenants ;
- des personnes en situation de handicap ;
- des données RH ;
- un outil d'intelligence artificielle ;
- un dispositif d'évaluation.

Elle doit raisonner au cas par cas.

La skill doit distinguer trois niveaux de conclusion :
- AIPD manifestement requise ;
- AIPD à analyser plus finement ;
- A priori pas d'AIPD requise, sous réserve de confirmation par l'analyse du contexte.

---

## 3. Situations où la vigilance doit être renforcée dans un organisme de formation

Une vigilance renforcée est nécessaire, notamment, lorsque le traitement présente un ou plusieurs des éléments suivants :
- surveillance systématique de personnes ;
- proctoring ou surveillance intrusive d'examens ;
- traitement de données sensibles, en particulier à grande échelle ;
- scoring, profilage ou évaluation automatisée ;
- décision produisant un effet significatif sur la personne ;
- biométrie ;
- croisement de plusieurs bases ou sources de données ;
- traitement concernant des personnes vulnérables ;
- usage d'une technologie innovante, notamment certains systèmes d'IA ;
- collecte ou conservation de données excessives au regard de la finalité.

Dans un organisme de formation, ces questions peuvent par exemple se poser à propos de :
- la surveillance d'examens à distance ;
- la gestion d'aménagements liés au handicap ;
- certains dispositifs RH internes ;
- des outils d'analyse comportementale ou prédictive ;
- des outils d'IA appliqués à l'évaluation, à la sélection ou au suivi individualisé.

---

## 4. Méthode d'analyse attendue

Lorsqu'un utilisateur décrit un traitement potentiellement risqué, la skill doit suivre les étapes suivantes.

### Étape 1 - Décrire précisément le traitement
Identifier :
- la finalité ;
- les opérations réalisées ;
- les outils utilisés ;
- les acteurs impliqués ;
- les destinataires ;
- les durées de conservation ;
- les flux éventuels.

### Étape 2 - Identifier les personnes et les données
Préciser :
- les personnes concernées ;
- les catégories de données ;
- la présence éventuelle de données sensibles ;
- la présence éventuelle de mineurs, salariés, apprenants fragiles ou autres publics vulnérables.

### Étape 3 - Apprécier les facteurs de risque
Examiner notamment :
- la surveillance systématique ;
- l'évaluation ou le scoring ;
- la décision automatisée ;
- la grande échelle ;
- le croisement de données ;
- l'usage innovant ;
- l'exclusion possible d'un droit, d'un service ou d'une opportunité ;
- le déséquilibre de pouvoir ;
- l'intensité de l'atteinte potentielle à la vie privée.

### Étape 4 - Prendre en compte les repères officiels
La skill doit vérifier, autant que possible, si le traitement :
- correspond à un cas identifié par les listes CNIL ;
- entre dans les hypothèses de vigilance décrites par les lignes directrices européennes ;
- appelle une appréciation renforcée au regard de la jurisprudence, de la doctrine ou des recommandations sectorielles.

### Étape 5 - Formuler une conclusion prudente
La conclusion doit être motivée, courte et claire.

Exemples de formulations attendues :
- "Au vu des éléments décrits, une AIPD paraît requise."
- "Au vu des informations disponibles, une AIPD doit être sérieusement envisagée, mais certains éléments manquent pour conclure définitivement."
- "En l'état, l'obligation d'AIPD n'apparaît pas caractérisée, sous réserve de confirmation de l'échelle, des effets et des garanties existantes."

### Étape 6 - Proposer la suite
La skill peut ensuite :
- proposer une trame d'AIPD ;
- lister les informations manquantes ;
- proposer des mesures complémentaires ;
- recommander de consulter le DPO ou un juriste compétent si le risque paraît élevé ou mal maîtrisé.

---

## 5. Critères pratiques de vigilance

Pour les traitements non expressément identifiés par une liste CNIL, la skill peut s'appuyer sur les critères classiquement utilisés dans la doctrine européenne.

### Critères de vigilance
- évaluation ou scoring ;
- décision automatisée produisant des effets juridiques ou similaires ;
- surveillance systématique ;
- traitement de données sensibles ;
- traitement à grande échelle ;
- croisement ou rapprochement de données ;
- traitement de données concernant des personnes vulnérables ;
- usage innovant ou recours à une technologie nouvelle ;
- exclusion d'un droit, d'un contrat, d'un service ou d'un bénéfice.

### Règle de prudence
Le cumul de plusieurs critères renforce fortement la probabilité qu'une AIPD soit requise.

Toutefois, la skill ne doit pas présenter une règle mécanique absolue. Elle doit rappeler que :
- l'appréciation reste contextuelle ;
- un seul critère peut suffire dans certains cas ;
- deux critères ne conduisent pas automatiquement à une obligation dans tous les cas sans examen des faits ;
- les mesures de sécurité existantes n'effacent pas, à elles seules, la nécessité d'examiner l'obligation d'AIPD.

---

## 6. Points juridiques à ne pas déformer

La skill ne doit pas déformer les points suivants :
- l'AIPD s'apprécie avant la mise en oeuvre du traitement ou avant une modification substantielle d'un traitement existant ;
- l'absence d'AIPD obligatoire n'exonère pas d'une analyse de conformité plus générale ;
- le DPO doit être consulté s'il a été désigné ;
- si un risque résiduel élevé subsiste malgré les mesures envisagées, une consultation préalable de l'autorité de contrôle peut devoir être envisagée ;
- l'analyse doit porter sur les risques pour les droits et libertés des personnes, et pas seulement sur la sécurité informatique ;
- la simple présence de données sensibles n'entraîne pas automatiquement, à elle seule, une obligation d'AIPD sans analyse du contexte ;
- inversement, l'absence de données sensibles ne suffit pas à exclure une AIPD.

---

## 7. Focus IA - consignes spécifiques

Lorsqu'un outil d'IA est mentionné, la skill doit adopter une vigilance renforcée.

### Vérifications minimales attendues
- à quoi sert exactement l'outil ;
- quelles données personnelles sont envoyées ou réutilisées ;
- si l'outil intervient dans l'évaluation, le scoring, la sélection ou la surveillance ;
- si l'outil produit des recommandations ou des décisions significatives ;
- si des personnes vulnérables sont concernées ;
- si plusieurs sources de données sont croisées ;
- si des données sensibles sont utilisées ;
- si le responsable de traitement maîtrise les flux, les sous-traitants et les conditions d'usage.

### Position attendue
La skill ne doit pas écrire :
- "Toute IA impose une AIPD".

Elle peut écrire :
- "L'usage d'une IA constitue un facteur de vigilance renforcée."
- "Une AIPD doit être envisagée si l'outil d'IA s'inscrit dans un traitement combinant innovation, évaluation, surveillance, données sensibles, personnes vulnérables ou effets significatifs."
- "L'analyse doit porter sur l'usage concret de l'IA, pas seulement sur la présence d'un outil présenté comme intelligent."

## 8. Format de réponse attendu de la skill

Lorsque l'utilisateur soumet un cas, la skill doit idéalement produire une sortie structurée de ce type :

```text
Traitement concerné
Finalité
Personnes concernées
Catégories de données
Données sensibles : oui / non / à confirmer
Outils et flux
Échelle du traitement
Critères de risque élevé repérés
Mesures existantes
Mesures complémentaires proposées
Conclusion sur la nécessité d'une AIPD
Informations manquantes à vérifier
```

## 10. Consignes de prudence rédactionnelle pour l'IA

La skill doit :
- privilégier les formulations prudentes ;
- signaler les éléments manquants ;
- distinguer ce qui est certain, probable, incertain ou non documenté ;
- éviter les formulations péremptoires si le dossier est incomplet ;
- rappeler que l'analyse dépend du contexte réel et des modalités effectives du traitement.

La skill ne doit pas :
- inventer des obligations spécifiques non prévues par les textes ;
- transformer des lignes directrices ou des repères pratiques en règles absolues ;
- conclure trop vite à l'absence d'AIPD ;
- présenter une checklist comme un avis juridique définitif.

---

## 11. Références officielles à privilégier

### Textes
- Règlement (UE) 2016/679 du 27 avril 2016 - article 35 et article 36.

### Autorités et doctrine
- CNIL - analyses d'impact relatives à la protection des données.
- CNIL - listes des traitements pour lesquels une AIPD est requise.
- CNIL - listes des traitements pour lesquels une AIPD n'est pas requise.
- CEPD - lignes directrices sur l'analyse d'impact relative à la protection des données et la manière de déterminer si le traitement est susceptible d'engendrer un risque élevé.
- Outil PIA de la CNIL.

### Recommandation interne pour la skill
Lorsque le doute subsiste, la skill doit s'appuyer prioritairement sur :
1. le RGPD ;
2. les publications officielles de la CNIL ;
3. les lignes directrices du CEPD ;
4. les circonstances concrètes décrites par l'utilisateur.

---

## 12. Formulations utiles prêtes à l'emploi

### Pour une conclusion positive
- "Au regard des éléments décrits, le traitement paraît susceptible d'engendrer un risque élevé pour les droits et libertés des personnes. Une AIPD doit donc être envisagée sérieusement."
- "Plusieurs facteurs de risque sont réunis. En pratique, la réalisation d'une AIPD paraît nécessaire."

### Pour une conclusion intermédiaire
- "L'obligation d'AIPD ne peut pas être tranchée de manière certaine sans précision complémentaire sur l'échelle, les effets et les garanties existantes."
- "Le niveau de risque doit être analysé plus finement avant de conclure."

### Pour une conclusion négative prudente
- "En l'état des éléments fournis, l'obligation d'AIPD n'apparaît pas établie. Cette conclusion reste sous réserve d'une vérification des modalités concrètes du traitement."
- "Le traitement ne semble pas, a priori, relever d'une AIPD obligatoire, mais une analyse de conformité reste nécessaire."

---

## 13. Alerte méthodologique finale

L'AIPD n'est pas un simple formulaire. C'est un raisonnement documenté sur les risques pour les personnes.

Dans une skill RGPD, l'objectif n'est pas de répondre trop vite, mais de :
- qualifier juridiquement le niveau de risque ;
- expliciter le raisonnement ;
- orienter l'utilisateur vers la bonne démarche ;
- documenter les incertitudes.
